# Obsidian 思考拡張プラグイン PoC — PLAN.md

## 背景・課題意識

### ベクトル検索RAGの本質的限界

ObsidianにMarkdownでノートを蓄積し、ベクトル検索でナレッジを引き出す構成が普及しているが、これには根本的な問題がある。

ベクトル検索が持ってくるのは「意味的近傍」、つまり同じ語彙圏・同じ抽象レベルの情報に限られる。ブレインストーミングや思考拡張で本当に価値があるのは「意味的には遠いが、構造的・パターン的に繋がっている情報」であり、ベクトル検索はこれを原理的に持ってこられない。

例：「免疫システムの自己/非自己認識」と「Gitのコンフリクト解決」は意味的に無関係だが、「差分検出と境界の設定」という構造が同型。

### 検索パラダイム自体の問題

「グラフDBにすれば解決」という方向も考えたが、より根本的な問題がある。

人間がブレインストーミングする際、記憶から「検索」しているのではなく、ある概念の活性化が連鎖的に別概念の想起を引き起こす（スプレッド・アクティベーション）。つまり問題は「グラフかベクトルか」ではなく、**静的な検索から動的な伝播モデルへの転換**が必要、ということ。

### 参考：Corpus2Skill（2026年4月）

ベクトルDBを使わず、ナレッジ全体を階層的ディレクトリ構造（目次）に変換し、LLMエージェントがツリーを辿って探索する手法。「網羅的に正しいものを持ってくる」精度向上が目的であり、今回の「思考を揺さぶる」目的とは異なるが、階層構造の隣枝に意図的にジャンプする設計として応用できる可能性がある。

## 欲しいもの

**「書いてる最中にシステムが割り込んで思考を揺さぶる」体験**

- 「知識を貯めて後で引く」ではなく、**書いている最中への介入**
- 答えを持ってくるのではなく、**問いや観察を出す**
- 書くリズムを壊さず、**視野の端に何かがいる**感覚
- 意図的に「逆・矛盾・隣の構造」を出して、書き手に摩擦を与える

### 具体的なUXイメージ

| トリガー | システムの反応 |
|---|---|
| 書いていて手が止まる | 同じ構造で過去に詰まった箇所を薄く表示 |
| 特定ワードを打った瞬間 | そのワードを過去に使った異なる文脈を一行で提示 |
| 既存ノートと矛盾する記述 | 「N年前に逆のことを書いている」とだけ表示 |
| 同じテーマへの繰り返し回帰 | 「このテーマに7回戻っている」というパターン提示 |

矛盾を解消させるのではなく、**矛盾を見せるだけ**。答えではなく問いを出す。

## アーキテクチャ設計（PoC）

### 全体フロー

```
Obsidianエディタ
  ↓ editor-changeイベント（プラグイン）
  ↓ デバウンス 3〜5秒
ローカルAPI（軽量embedding）
  ↓ 過去ノート群から候補20本に絞る
Ollama / qwen3.6-64k（MoE, num_ctx=65536, host: ws via Tailscale）
  ↓ 候補20本 + 現在の編集テキストを渡す
  ↓ 構造的類似・矛盾・繰り返しパターンを推論
  ↓ 「問いか一行の観察」のみ返す（プロンプトで縛る）
Obsidianサイドパネル or ポップオーバー
  ↓ 邪魔しない形で表示
```

### ローカル環境

- 推論モデル（一次）: qwen3.6-64k（Qwen3.5 MoE, 36B総 / ~3Bアクティブ, Q4_K_M, 23GB, num_ctx=65536, 256K上限, thinking対応）
  - 速度優先。書き手のフローを壊さないレイテンシが PoC の本質的価値
- 推論モデル（fallback）: qwen3.6-27b-128k（Dense, 27B, Q4_K_M, 17GB）
  - 構造的同型の発見が MoE で甘い場合に切替（Settings から model 名変更のみ）
- 埋め込みモデル: bge-m3:567m（多言語, 8K context, 1.2GB, 1024-dim）
  - 候補絞り込み用。日本語ノート向けに選定
- ホスト: ws（Ubuntu on WSL2 / RTX 5090）。Mac から Tailscale 越しに `http://100.92.242.85:11434` を叩く
- Obsidianプラグイン: TypeScript（Mac 上で開発）

これだけ。通知なし、押しつけなし。

## LLMへのプロンプト設計方針

- 出力は必ず**一行**に縛る
- 「答え」ではなく「問い」か「観察」の形式のみ許可
- 類似より**対立・矛盾・構造的同型**を優先して探す
- 「このノートと最も関連が深いもの」ではなく「このノートが書き手の視野を広げるために今見るべきもの」という指示

例：
```
以下のテキストを読み、過去ノート候補の中から
「意味は遠いが構造が似ている」か「矛盾している」ものを一つ選び、
なぜそれを選んだかを一行の問いの形で出力せよ。
答えを出してはならない。
```

## 今後の拡張アイデア（PoC後）

- 抽象度の強制シフト：「一段上の抽象概念は何か」「一段下の具体例は何か」
- 時間軸の導入：「この考えの起源ノート」「これが進化した先のノート」
- 問いの逆生成：そのノートが答えになるような問いを生成し、別ドメインで同じ問いに答えているノートを引く
- 認知パターンの可視化：「あなたはいつも境界の話をしている」「この3つの問いは実は同じかもしれない」
